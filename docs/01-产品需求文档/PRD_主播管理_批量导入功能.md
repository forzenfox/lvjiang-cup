## 2. 背景与目标

### 2.1 背景

当前项目已具备主播管理基础功能，管理员可通过后台页面进行主播的增删改查及拖拽排序。但在实际运营中，赛事活动往往涉及大量主播（如驴酱杯赛事通常有数十位主播参与），逐一手动添加效率极低且容易出错。

### 2.2 目标

- 支持通过 Excel 模板批量导入主播信息，提升运营效率
- 保持与现有主播管理页面（`AdminStreamers`）及 API 的兼容性
- 复用项目已有的导入功能架构（参考战队导入实现），降低开发成本
- 提供完善的错误提示与导入结果反馈

### 2.3 成功指标

- 单次导入支持 100+ 条主播数据
- 导入成功率 ≥ 99%（数据格式正确前提下）
- 导入操作平均耗时 < 5 秒（100 条数据）
- 运营人员无需开发介入即可完成批量导入

***

## 3. 范围

### 3.1 In Scope

- 主播信息批量导入（Excel .xlsx 格式）
- 导入模板下载功能
- 导入数据预览与校验
- 导入结果统计与错误报告下载
- 与现有主播管理页面的集成

### 3.2 Out of Scope

- 从其他平台（如斗鱼 API）自动同步主播信息
- 图片资源的自动下载与上传（外链 URL 保持原样导入，需人工后续处理）
- 主播信息的批量导出功能（后续迭代可考虑）
- 导入历史记录与回滚功能

## 5. 需求详情

### 5.1 功能需求

#### FR-1: 导入模板下载

- 在主播管理页面提供「下载导入模板」按钮
- 模板文件为 `.xlsx` 格式，包含表头、示例数据及数据验证规则
- 模板字段与现有主播数据模型保持一致

#### FR-2: Excel 文件上传与解析

- 支持拖拽上传或点击选择文件
- 仅接受 `.xlsx` 格式，文件大小限制 5MB
- 前端基础校验：文件格式、大小

#### FR-3: 数据校验规则

| 字段     | 必填 | 校验规则               | 说明                       |
| ------ | -- | ------------------ | ------------------------ |
| 主播类型   | 是  | 枚举：`驴酱主播` / `嘉宾主播` | 映射为 `internal` / `guest` |
| 主播昵称   | 是  | 非空，长度 1-50 字符      | 唯一性由业务逻辑控制               |
| 海报 URL | 否  | 合法 URL 格式或空        | 支持 http/https 链接         |
| 直播间号   | 否  | 纯数字或合法 URL         | 纯数字自动拼接为斗鱼完整 URL         |
| 个人简介   | 否  | 长度 0-500 字符        | <br />                   |

#### FR-4: 导入模式

- **全量覆盖模式**（第一版唯一模式）：导入前删除所有现有主播数据，根据 Excel 内容重新创建全部主播记录，所有 ID 和 `sort_order` 重新生成
- ⚠️ **重要警告**：此操作将永久删除现有所有主播数据，且无法恢复。导入后已有引用（如对战数据关联）将失效，需要重新关联。
- 导入完成后自动刷新主播列表

#### FR-5: 导入结果反馈

- 显示导入统计：总计 / 新增 / 更新 / 失败
- 失败数据展示错误原因（行号、字段、错误描述）
- 支持下载错误报告（文本格式）

#### FR-6: 主播列表刷新

- 导入成功后自动刷新 `AdminStreamers` 页面列表
- 保持现有排序规则（`sort_order ASC, created_at DESC`）

### 5.2 非功能需求

| 需求项 | 要求                           |
| --- | ---------------------------- |
| 性能  | 100 条数据导入耗时 < 5 秒            |
| 并发  | 单次仅允许一个导入任务执行                |
| 安全  | 仅管理员可访问（复用现有 `JwtAuthGuard`） |
| 兼容性 | 支持 Windows/Mac 生成的 Excel 文件  |
| 可靠性 | 导入失败时数据库事务回滚，保证数据一致性         |

## 8. API 接口设计

### 8.1 下载导入模板

```http
GET /admin/streamers/import/template
```

- **认证**: 需要 (`JwtAuthGuard`)
- **响应**: `Blob` (Excel 文件)

### 8.2 批量导入主播

```http
POST /admin/streamers/import
Content-Type: multipart/form-data
```

**请求参数**:

| 参数   | 类型   | 必填 | 说明               |
| ---- | ---- | -- | ---------------- |
| file | File | 是  | Excel 文件 (.xlsx) |

**响应结构**:

```typescript
interface StreamerImportResult {
  total: number;      // 处理的总行数
  created: number;    // 成功创建的主播数（全量覆盖模式下等于成功数）
  failed: number;     // 失败数
  errors?: StreamerImportError[];
  externalUrlItems?: string[]; // 外链海报URL提醒
}

interface StreamerImportError {
  row: number;        // Excel 行号
  nickname: string;   // 主播昵称
  field: string;      // 错误字段
  message: string;    // 错误描述
}
```

### 8.3 下载错误报告

```http
POST /admin/streamers/import/error-report
Content-Type: application/json
```

**请求体**:

```json
{
  "errors": [
    { "row": 3, "nickname": "", "field": "nickname", "message": "主播昵称不能为空" }
  ]
}
```

**响应**: `Blob` (文本文件)

## 9. 技术实现方案

### 9.1 后端实现

#### 9.1.1 新增模块文件

```
backend/src/modules/streamers/
├── controllers/
│   └── streamers-import.controller.ts    # 导入控制器
├── services/
│   └── streamers-import.service.ts       # 导入服务
├── dto/
│   └── streamer-import.dto.ts            # 导入相关 DTO
└── utils/
    └── streamer-excel.util.ts            # Excel 解析工具
```

#### 9.1.2 核心逻辑

1. **模板生成**: 使用 `exceljs` 生成标准模板，包含表头、示例数据、数据验证（下拉列表）
2. **Excel 解析**: 读取上传文件，解析为 `CreateStreamerDto[]`
3. **数据校验**:
   - 必填字段检查
   - 主播类型有效性检查
   - 直播间号格式处理
4. **数据库操作**（全量覆盖模式）:
   - 开启事务
   - 删除所有现有主播数据（`DELETE FROM streamers`）
   - 遍历 Excel 数据，逐条 `INSERT` 新记录（生成新 ID，按顺序分配 `sort_order`）
   - 提交事务
5. **缓存清理**: 导入完成后清除主播相关缓存

#### 9.1.3 关键代码参考

复用现有 `StreamersService` 的 `create` / `update` 逻辑，以及 `TeamsImportService` 的事务处理模式。

### 9.2 前端实现

#### 9.2.1 新增/修改文件

```
frontend/src/
├── api/
│   └── streamers-import.ts               # 导入相关 API 封装
├── components/import/
│   └── StreamerImportDialog.tsx          # 主播导入弹窗（参考 ImportDialog）
└── pages/admin/
    └── Streamers.tsx                     # 新增「批量导入」按钮入口
```

#### 9.2.2 组件设计

- **StreamerImportDialog**: 复用现有 `ImportDialog` 的 UI 风格与交互逻辑
  - 支持文件拖拽上传
  - 导入说明展示
  - 模板下载按钮
  - 导入结果展示

#### 9.2.3 API 封装 (`streamers-import.ts`)

参考 `teams-import.ts` 实现：

- `downloadTemplate()`: 下载模板
- `importStreamers(file: File)`: 执行导入
- `downloadErrorReport(errors)`: 下载错误报告

## 10. 错误处理

### 10.1 前端错误

| 场景              | 处理方式                       |
| --------------- | -------------------------- |
| 文件格式错误（非 .xlsx） | 提示「仅支持 .xlsx 格式的 Excel 文件」 |
| 文件大小超过 5MB      | 提示「文件大小不能超过 5MB」           |
| 网络异常            | 提示「导入失败，请检查网络后重试」          |
| 导入部分失败          | 展示失败详情，提供错误报告下载            |

### 10.2 后端错误

| 场景       | 处理方式                      |
| -------- | ------------------------- |
| 缺少必要列头   | 返回 400，提示缺少的列名            |
| 数据校验失败   | 记录错误明细，继续处理其他行，最终返回部分成功结果 |
| 数据库异常    | 事务回滚，返回 500 错误            |
| 空文件/无数据行 | 返回 400，提示「未检测到有效数据」       |

## 12. 风险评估

| 风险            | 影响 | 概率 | 缓解措施                                |
| ------------- | -- | -- | ----------------------------------- |
| Excel 格式兼容性问题 | 中  | 中  | 使用 `exceljs` 库，支持主流 Excel 版本；提供标准模板 |
| 大量数据导入超时      | 中  | 低  | 后端异步处理或分页导入；设置合理的文件大小限制             |
| 外链海报 URL 失效   | 低  | 高  | 导入时仅做格式校验，不做可达性检查；运营人员自行确认          |
| 误操作覆盖已有数据     | 高  | 中  | 弹窗内显式警告说明；全量覆盖模式为唯一选项，用户必须明确知晓风险后操作 |
