# 首页数据加载策略优化方案

## 1. 现状分析

### 1.1 当前数据加载架构

主页面 `Home.tsx` 在加载时会触发多个 API 请求，同时各子组件也独立发起请求：

| 请求来源 | API 端点 | 触发时机 | 数据变化频率 |
|---------|---------|---------|------------|
| Home.tsx `loadAllData` | `GET /streams/active` | 首屏优先级1 | **高（直播状态实时变化）** |
| Home.tsx `loadAllData` | `GET /teams` | 首屏优先级2 | **低（战队信息相对固定）** |
| Home.tsx `loadAllData` | `GET /matches` | 后台并行 | **中（赛程/比分可能更新）** |
| Home.tsx `loadAllData` | `GET /videos` | 后台并行 | **低（视频列表相对固定）** |
| HeroSection | `GET /streams/active` | 组件挂载 | 重复请求 |
| TeamSection | `GET /teams` | 组件挂载 | 重复请求 |
| ScheduleSection | `GET /matches` + `GET /teams` | 组件挂载 | 重复请求 |
| StreamerSection | `GET /streamers` | 组件挂载 + 每30秒轮询 | **低（主播信息相对固定）** |

### 1.2 已识别的问题

#### 问题1：严重的重复请求（高优先级）

Home.tsx 和子组件各自独立请求相同数据：

- **直播数据**：Home.tsx + HeroSection → 2次相同请求
- **战队数据**：Home.tsx + TeamSection + ScheduleSection → 3次相同请求
- **比赛数据**：Home.tsx + ScheduleSection → 2次相同请求

虽然前端 `requestCache` 有短期缓存（15-60秒），但页面首次加载时这些请求几乎同时发出，缓存尚未生效，导致同一数据被请求2-3次。

#### 问题2：HomeDataContext 数据未被消费且不更新

- `HeroSection`、`TeamSection`、`ScheduleSection` 都没有使用 `useHomeData()`，各自独立请求数据
- `HomeDataContext.tsx` 使用 `const [data] = useState(initialData)`，初始值只在首次渲染时生效，后续 `initialData` 变化不会更新 `data`

#### 问题3：首屏串行加载

stream 和 teams 是串行加载的，但它们之间没有依赖关系。首屏渲染需要等待两个请求串行完成，延迟是两者之和。

#### 问题4：StreamerSection 不必要的轮询

主播信息变化频率极低，每30秒轮询完全没有必要，白白消耗服务器资源。

#### 问题5：ScheduleSection Tab 切换重复请求

切换瑞士轮/淘汰赛 Tab 时会重新请求所有数据，但数据实际上没有变化。

#### 问题6：全量预加载策略

当前 Home.tsx 在页面加载时一次性请求所有模块的数据（直播、战队、比赛、视频），但用户可能只浏览部分区域。非可视区域的数据请求是浪费。

### 1.3 已有的优化措施

1. **React.lazy 懒加载**：非首屏组件使用 `lazy()` + `Suspense` 延迟加载组件代码
2. **前端请求缓存**：`requestCache` 提供 TTL 缓存（stream 15s, matches 30s, teams 60s, videos 120s）
3. **后端服务缓存**：`BaseCachedService` + `CacheService`（NodeCache，默认60s TTL）
4. **图片懒加载**：`loading="lazy"` + `decoding="async"`
5. **CDN 图片降级**：HeroSection 的 CDN → 本地降级策略

## 2. 业务场景数据变化频率分析

根据实际业务场景，首页各模块数据的变化频率差异很大：

| 数据类型 | 变化频率 | 典型场景 | 合理缓存时长 | 是否需要轮询 |
|---------|---------|---------|------------|------------|
| 直播信息 | **高** | 开播/下播状态变化、直播标题更新 | 10-15秒 | ✅ 是（可视区域内） |
| 赛程/比赛 | **中** | 比赛开始、比分更新、状态变化 | 30-60秒 | ✅ 是（可视区域内） |
| 战队信息 | **低** | 战队创建/编辑/队员调整（管理后台操作） | 5分钟 | ❌ 否 |
| 主播信息 | **低** | 主播添加/编辑（管理后台操作） | 5分钟 | ❌ 否 |
| 视频列表 | **低** | 视频添加/排序/启用禁用（管理后台操作） | 5分钟 | ❌ 否 |
| 特别鸣谢 | **极低** | 赞助商/工作人员变更（几乎不变） | 永久（构建时注入） | ❌ 否 |

**核心洞察**：低频变化数据无需轮询，只需在首次访问时加载一次，配合合理的缓存策略即可。只有直播信息和赛程信息需要定期刷新。

## 3. 优化方案设计

### 3.1 核心原则

1. **按需加载**：用户访问什么模块就加载对应模块的数据，不再全量预加载
2. **按频率缓存**：根据数据变化频率设计差异化的缓存策略
3. **消除重复**：同一数据在页面内只请求一次，通过 Context 共享
4. **智能刷新**：只有高频变化数据才在可视区域内轮询刷新

### 3.2 按需加载架构设计

#### 当前架构（全量预加载）

```
Home.tsx 挂载
  ├── loadAllData() 一次性加载所有数据
  │   ├── streamService.get()      ← 立即请求
  │   ├── teamService.getAll()     ← 立即请求
  │   ├── matchService.getAll()    ← 立即请求
  │   └── videoApi.getVideos()     ← 立即请求
  ├── HeroSection 挂载 → streamService.get()      ← 重复请求
  ├── VideoCarousel 挂载 → 使用 Home 传入的 videos
  ├── StreamerSection 挂载 → streamersApi.getAll() ← 额外请求 + 30s轮询
  ├── TeamSection 挂载 → teamService.getAll()      ← 重复请求
  ├── ScheduleSection 挂载 → matchService.getAll() + teamService.getAll() ← 重复请求
  └── ThanksSection 挂载 → 使用 window.THANKS_DATA（无请求）
```

#### 优化后架构（按需加载 + Context 共享）

```
Home.tsx 挂载
  ├── 不再统一 loadAllData()
  ├── HomeDataProvider 提供 Context（数据为空，按需填充）
  │
  ├── HeroSection 挂载 → useHomeData().stream
  │   └── 首次访问时请求 stream，缓存到 Context
  │   └── 可视区域内每15秒刷新直播状态
  │
  ├── LazyVideoCarousel 挂载 → useHomeData().videos
  │   └── 进入视口时请求 videos，缓存到 Context
  │   └── 无轮询（低频变化数据）
  │
  ├── StreamerSection 挂载 → useHomeData().streamers
  │   └── 进入视口时请求 streamers，缓存到 Context
  │   └── 无轮询（低频变化数据）
  │
  ├── TeamSection 挂载 → useHomeData().teams
  │   └── 进入视口时请求 teams，缓存到 Context
  │   └── 无轮询（低频变化数据）
  │
  ├── ScheduleSection 挂载 → useHomeData().matches + useHomeData().teams
  │   └── 进入视口时请求 matches，复用 Context 中的 teams
  │   └── 可视区域内每60秒刷新赛程
  │   └── Tab 切换不触发重新请求
  │
  └── ThanksSection 挂载 → window.THANKS_DATA（无请求）
```

### 3.3 差异化缓存策略

#### 前端缓存（requestCache）调整

```typescript
// 当前缓存时长

  teams: 60_000,      // 60秒
  matches: 30_000,    // 30秒
  stream: 15_000,     // 15秒
  videos: 120_000,    // 120秒

  teams: 300_000,     // 5分钟（低频变化）
  matches: 60_000,    // 1分钟（中频变化）
  stream: 15_000,     // 15秒（高频变化）
  videos: 300_000,    // 5分钟（低频变化）

```

#### 后端缓存（CacheService）调整

后端 `CacheService` 默认 TTL 为60秒，建议针对不同服务设置差异化 TTL：

| 服务 | 当前 TTL | 建议 TTL | 说明 |
|------|---------|---------|------|
| TeamsService | 60s | 300s | 战队信息低频变化 |
| MatchesService | 60s | 60s | 保持不变 |
| StreamsService | 60s | 15s | 直播状态需要实时性 |
| VideosService | 60s | 300s | 视频列表低频变化 |
| StreamersService | 60s | 300s | 主播信息低频变化 |

### 3.4 智能刷新策略

仅对高频变化数据实施可视区域内的定时刷新：

```typescript
// 可视区域感知的刷新 Hook
function useVisibleRefresh(
  fetchFn: () => Promise<void>,
  intervalMs: number,
  options?: { enabled?: boolean }
) {
  const ref = useRef<HTMLElement>(null);
  const isVisible = useIntersectionObserver(ref);

  useEffect(() => {
    // 不可见或未启用时，不启动定时器
    if (!isVisible || !options?.enabled) return;

    const timer = setInterval(fetchFn, intervalMs);
    return () => clearInterval(timer);
  }, [isVisible, fetchFn, intervalMs, options?.enabled]);

  // 页面可见性变化时处理
  useEffect(() => {
    const handleVisibilityChange = () => {
      if (document.visibilityState === 'visible' && isVisible) {
        fetchFn(); // 页面恢复可见时刷新一次
      }
    };
    document.addEventListener('visibilitychange', handleVisibilityChange);
    return () => document.removeEventListener('visibilitychange', handleVisibilityChange);
  }, [fetchFn, isVisible]);

  return ref;
}
```

各模块刷新配置：

| 模块 | 是否定时刷新 | 刷新间隔 | 触发条件 |
|------|------------|---------|---------|
| HeroSection（直播） | ✅ 是 | 15秒 | 组件在可视区域内 |
| ScheduleSection（赛程） | ✅ 是 | 60秒 | 组件在可视区域内 |
| StreamerSection（主播） | ❌ 否 | - | 仅首次加载 + 手动刷新 |
| TeamSection（战队） | ❌ 否 | - | 仅首次加载 + 手动刷新 |
| VideoCarousel（视频） | ❌ 否 | - | 仅首次加载 + 手动刷新 |

## 4. 详细实现方案

### 4.1 重构 HomeDataContext（核心改动）

将 HomeDataContext 从"被动接收数据"改为"主动按需加载"：

```typescript
// HomeDataContext.tsx 重构设计
interface HomeDataContextValue {
  // 数据
  stream: Stream | null;
  teams: ApiTeam[];
  matches: unknown[];
  videos: VideoItem[];
  streamers: Streamer[];

  // 加载状态
  isLoading: Record<string, boolean>;

  // 按需加载方法（首次调用时请求，后续直接返回缓存）
  fetchStream: () => Promise<void>;
  fetchTeams: () => Promise<void>;
  fetchMatches: () => Promise<void>;
  fetchVideos: () => Promise<void>;
  fetchStreamers: () => Promise<void>;

  // 强制刷新方法（用户手动触发）
  refresh: (module: string) => Promise<void>;
}
```

关键设计：

1. **懒加载方法**：`fetchStream()` 等方法在首次调用时才发起请求，后续调用直接返回缓存数据
2. **数据存储在 Context 中**：所有子组件通过 Context 共享数据，避免重复请求
3. **加载状态追踪**：每个模块独立的 loading 状态，支持骨架屏展示

### 4.4 IntersectionObserver Hook 实现

用于实现"进入视口时才加载数据"：

```typescript
// hooks/useIntersectionObserver.ts
function useIntersectionObserver(
  ref: React.RefObject<HTMLElement>,
  options?: IntersectionObserverInit
): boolean {
  const [isVisible, setIsVisible] = useState(false);

  useEffect(() => {
    const element = ref.current;
    if (!element) return;

    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting) {
          setIsVisible(true);
          // 一旦可见就不再观察（数据只需加载一次）
          observer.unobserve(element);
        }
      },
      { threshold: 0.1, ...options }
    );

    observer.observe(element);
    return () => observer.disconnect();
  }, [ref, options]);

  return isVisible;
}
```

## 5. 优化效果预估

### 5.1 请求量对比

| 指标 | 当前 | 优化后 | 改善幅度 |
|------|------|--------|---------|
| 首页首次加载 API 请求数 | 7-8 | 2-3（仅首屏可见区域） | ↓ 60-75% |
| 重复请求 | stream×2, teams×3, matches×2 | 各×1 | ↓ 100% |
| 后台轮询请求/分钟 | 2次（streamer 30s轮询） | 0次 | ↓ 100% |
| 非可视区域请求 | 4-5个 | 0个 | ↓ 100% |

### 5.2 首屏加载时间对比

| 场景 | 当前 | 优化后 | 说明 |
|------|------|--------|------|
| 首屏可见区域 | T_stream + T_teams + T_matches + T_videos | T_stream（仅直播） | 首屏只需直播数据 |
| 滚动到视频区域 | 已预加载 | T_videos | 按需加载，延迟可接受 |
| 滚动到主播区域 | 已预加载 | T_streamers | 按需加载，延迟可接受 |
| 滚动到战队区域 | 已预加载 | T_teams | 按需加载，延迟可接受 |
| 滚动到赛程区域 | 已预加载 | T_matches（teams 已缓存） | 按需加载，teams 复用缓存 |

### 5.3 服务器负载对比

| 指标 | 当前 | 优化后 | 改善幅度 |
|------|------|--------|---------|
| 每用户每分钟 API 请求 | ~10次 | ~2-4次 | ↓ 60-80% |
| 数据库查询次数/分钟 | ~20次（含重复） | ~4次 | ↓ 80% |
| 后端缓存命中率 | 中 | 高（TTL 延长） | ↑ 50%+ |

## 6. 实施计划

### 阶段1：消除重复请求（优先级：高，影响最大）

1. 重构 `HomeDataContext`，支持按需加载和数据共享
2. 修改 `HeroSection`、`TeamSection`、`ScheduleSection` 使用 Context
3. 移除 `Home.tsx` 中的 `loadAllData` 统一加载逻辑

### 阶段2：按需加载（优先级：高）

1. 实现 `useIntersectionObserver` Hook
2. 各模块在进入视口时才触发数据请求
3. 首屏仅加载 HeroSection 所需的直播数据

### 阶段3：差异化缓存和智能刷新（优先级：中）

1. 调整前端 `requestCache` 的 TTL 配置
2. 移除 StreamerSection 的 30 秒轮询
3. 实现 `useVisibleRefresh` Hook
4. 仅对直播（15s）和赛程（60s）实施可视区域定时刷新
5. 移除 ScheduleSection Tab 切换时的重复请求

### 阶段4：后端缓存优化（优先级：低，需后端配合）

1. 为不同服务配置差异化缓存 TTL
2. 考虑新增聚合 API（可选，视前端优化效果决定）

## 7. 风险与注意事项

1. **骨架屏体验**：按需加载意味着用户滚动到新区域时会有短暂的加载状态，需要确保骨架屏过渡流畅
2. **SEO 影响**：按需加载对 SEO 无影响，因为当前是 SPA 应用，搜索引擎爬虫本身无法执行 JS
3. **数据一致性**：Context 中的数据需要确保在用户长时间停留在页面时仍然有效，可通过手动刷新按钮兜底
4. **错误隔离**：各模块独立加载，某个模块请求失败不应影响其他模块的展示
5. **缓存失效**：管理后台修改数据后，前端缓存可能未及时更新，可通过缩短 TTL 或手动刷新解决
