## 1. 测试概述

### 1.1 项目背景

驴酱杯 LOL 娱乐赛事网站是一个仿照 LPL 官方赛事网站的单页面滚动式网站，为斗鱼驴酱公会的主播和水友提供赛事信息展示平台。项目采用 React + TypeScript 前端 + NestJS 后端，使用 SQLite 数据库，部署在 2C2G 服务器。

### 1.2 项目特点

| 特点 | 说明 |
|------|------|
| **前端技术栈** | React 18.3.1 + TypeScript + Vite + Zustand + Tailwind CSS |
| **后端技术栈** | NestJS 10.x + SQLite + JWT + node-cache |
| **数据持久化** | 前端：localStorage（Mock 数据服务）; 后端：SQLite |
| **部署方式** | 前端：Nginx; 后端：2C2G 服务器 + Docker |
| **用户角色** | 游客（查看）、管理员（管理后台） |
| **核心功能** | 赛事展示、战队管理、赛程管理、直播配置、瑞士轮/淘汰赛可视化 |

### 1.3 测试目标

1. **功能正确性**：确保所有功能模块按 PRD 要求正常工作
2. **数据一致性**：确保 localStorage/SQLite 数据读写正确，特别是晋级名单同步
3. **用户体验**：确保界面交互流畅，响应式设计正常
4. **回归测试**：防止已有功能在新代码提交后出现回退
5. **缺陷预防**：通过边界和异常测试发现潜在问题
6. **后端稳定性**：确保 API 接口稳定、数据库操作正确、缓存策略有效

### 1.4 测试范围

**纳入测试范围**：
- ✅ 前端所有页面和组件（首页、管理后台）
- ✅ 状态管理（Zustand store）
- ✅ Mock 数据服务
- ✅ 路由导航
- ✅ 用户交互（表单、按钮、Tab 切换等）
- ✅ 数据持久化（localStorage）
- ✅ 响应式布局
- ✅ 瑞士轮可视化编辑器
- ✅ 晋级名单管理功能
- ✅ 后端 API 接口测试
- ✅ 后端业务逻辑测试
- ✅ 数据库操作测试
- ✅ 缓存服务测试
- ✅ 认证授权测试
- ✅ 前后端集成测试

**不纳入测试范围**：
- ❌ 性能压力测试（个人项目，访问量低）
- ❌ 安全渗透测试（需专业工具）
- ❌ 兼容性测试（仅需支持现代浏览器）

## 2. 测试策略

### 2.1 测试层次

采用三层测试金字塔模型：

```
        ╱╲
       ╱  ╲
      ╱ E2E ╲      ← 关键路径 E2E 测试（AI 通过 DevTools/Playwright 执行）
     ╱────────╲
    ╱Integration╲   ← 模块集成测试（数据流、组件交互、API 集成）
   ╱──────────────╲
  ╱    Unit Tests  ╲ ← 大量单元测试（组件、工具函数、store、Service）
 ╱──────────────────╲
```

### 2.2 测试类型

| 测试类型 | 执行方式 | 工具 | 频率 |
|---------|---------|------|------|
| **单元测试** | 自动化 | Vitest (前端) + Jest (后端) | 每次代码变更 |
| **集成测试** | 自动化 | Vitest (前端) + Jest (后端) | 每次代码变更 |
| **E2E 测试** | AI 手动执行 + Playwright 自动化 | Chrome DevTools MCP + Playwright | 版本发布前 |
| **回归测试** | 自动化 + AI | Vitest + DevTools | 每次 bug 修复后 |
| **API 测试** | 自动化 | Supertest + Jest | 每次代码变更 |

### 2.3 测试优先级

**P0 - 关键路径（必须测试）**：
1. 首页加载和导航
2. 管理后台登录

5. 晋级名单管理和同步
6. 直播配置
7. 后端 API 认证授权

**P1 - 重要功能（应该测试）**：

## 7. 缺陷管理

### 7.1 缺陷分级标准

| 级别 | 名称 | 定义 | 响应时间 |
|------|------|------|---------|
| **P0** | 致命缺陷 | 系统崩溃、数据丢失、核心功能完全不可用 | 立即修复 |
| **P1** | 严重缺陷 | 主要功能异常、数据不一致、影响大部分用户 | 24 小时内 |
| **P2** | 一般缺陷 | 次要功能异常、UI 问题、不影响核心功能 | 本周内 |
| **P3** | 轻微缺陷 | 建议改进、文案错误、视觉优化 | 排期处理 |

```
发现缺陷 → 记录缺陷 → 分级评估 → 分配修复 → 修复验证 → 关闭缺陷
```
