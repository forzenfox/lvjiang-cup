## 1. 测试概述

### 1.1 项目背景

驴酱杯 LOL 娱乐赛事网站是一个仿照 LPL 官方赛事网站的单页面滚动式网站，为斗鱼驴酱公会的主播和水友提供赛事信息展示平台。项目采用 React + TypeScript 前端，使用 Mock 数据服务（localStorage 持久化），部署在 GitHub Pages。

### 1.2 项目特点

| 特点 | 说明 |
|------|------|
| **技术栈** | React 18.3.1 + TypeScript + Vite + Zustand + Tailwind CSS |
| **数据持久化** | localStorage（Mock 数据服务） |
| **部署方式** | GitHub Pages（静态站点） |
| **用户角色** | 游客（查看）、管理员（管理后台） |
| **核心功能** | 赛事展示、战队管理、赛程管理、直播配置、瑞士轮/淘汰赛可视化 |

### 1.3 测试目标

1. **功能正确性**：确保所有功能模块按 PRD 要求正常工作
2. **数据一致性**：确保 localStorage 数据读写正确，特别是晋级名单同步
3. **用户体验**：确保界面交互流畅，响应式设计正常
4. **回归测试**：防止已有功能在新代码提交后出现回退
5. **缺陷预防**：通过边界和异常测试发现潜在问题

### 1.4 测试范围

**纳入测试范围**：
- ✅ 前端所有页面和组件（首页、管理后台）
- ✅ 状态管理（Zustand store）
- ✅ Mock 数据服务
- ✅ 路由导航
- ✅ 用户交互（表单、按钮、Tab 切换等）
- ✅ 数据持久化（localStorage）
- ✅ 响应式布局

**不纳入测试范围**：
- ❌ 后端 API 测试（当前版本无真实后端）
- ❌ 性能压力测试（个人项目，访问量低）
- ❌ 安全测试（无真实用户认证）
- ❌ 兼容性测试（仅需支持现代浏览器）

## 2. 测试策略

### 2.1 测试层次

采用三层测试金字塔模型：

```
        ╱╲
       ╱  ╲
      ╱ E2E ╲      ← 少量关键路径 E2E 测试（AI 通过 DevTools 执行）
     ╱────────╲
    ╱Integration╲   ← 模块集成测试（数据流、组件交互）
   ╱──────────────╲
  ╱    Unit Tests  ╲ ← 大量单元测试（组件、工具函数、store）
 ╱──────────────────╲
```

### 2.2 测试类型

| 测试类型 | 执行方式 | 工具 | 频率 |
|---------|---------|------|------|
| **单元测试** | 自动化 | Vitest + Testing Library | 每次代码变更 |
| **集成测试** | 自动化 | Vitest + Testing Library | 每次代码变更 |
| **E2E 测试** | AI 手动执行 | Chrome DevTools MCP | 版本发布前 |
| **回归测试** | 自动化 + AI | Vitest + DevTools | 每次 bug 修复后 |

## 4. 测试范围与工具

### 4.1 单元测试策略

#### 4.1.1 测试范围

| 模块 | 测试对象 | 覆盖率目标 |
|------|---------|-----------|
| **Components** | 所有功能组件（SwissStage, EliminationStage, TeamSection 等） | 80% 分支覆盖 |
| **Store** | Zustand stores（advancementStore） | 100% 语句覆盖 |
| **Utils** | 工具函数（datetime.ts, utils.ts） | 100% 分支覆盖 |
| **Mock Service** | Mock 数据服务（service.ts） | 90% 语句覆盖 |
| **Pages** | 管理后台页面（Dashboard, Teams, Schedule 等） | 70% 语句覆盖 |

#### 4.1.2 工具选择

**测试框架**：Vitest
- 与 Vite 无缝集成
- 支持 TypeScript
- 快速的并行测试执行
- 内置覆盖率统计

**测试工具**：Testing Library
- React 组件测试最佳实践
- 用户视角的测试方法
- 自动清理机制

**断言库**：Vitest 内置 + jest-dom
- 丰富的 DOM 断言
- 可读性强的错误信息

#### 4.1.3 覆盖率目标

| 指标 | 目标值 | 说明 |
|------|-------|------|
| **语句覆盖率** | ≥85% | 所有代码行的执行比例 |
| **分支覆盖率** | ≥75% | 条件分支的覆盖比例 |
| **函数覆盖率** | ≥90% | 函数/方法的调用比例 |
| **行覆盖率** | ≥85% | 代码行的覆盖比例 |

**关键模块要求 100% 覆盖**：
- `src/store/advancementStore.ts`
- `src/mock/service.ts`
- `src/utils/datetime.ts`

### 4.2 集成测试计划

#### 4.2.1 模块交互测试

| 测试场景 | 涉及模块 | 测试点 |
|---------|---------|-------|
| **数据流测试** | MockService → Components → Store | 数据加载、更新、持久化 |
| **组件通信** | Parent → Child Props | Props 传递、回调函数 |
| **状态同步** | Store → Multiple Components | 状态更新、组件重渲染 |
| **路由导航** | React Router → Pages | 路由切换、参数传递 |

#### 4.2.2 接口测试规范

由于项目使用 Mock 服务，接口测试针对 Mock Service：

| 接口 | 测试内容 | 验证点 |
|------|---------|-------|
| `getTeams()` | GET 请求 | 返回战队列表、延迟模拟 |
| `updateTeam()` | PUT 请求 | 数据更新、localStorage 持久化 |
| `addTeam()` | POST 请求 | 新数据添加、ID 生成 |
| `deleteTeam()` | DELETE 请求 | 数据删除、关联处理 |
| `getMatches()` | GET 请求 | 比赛列表、队伍数据 enrich |
| `updateMatch()` | PUT 请求 | 比分更新、胜者判定 |
| `resetAllData()` | POST 请求 | 数据重置、store 清理 |
| `clearAllData()` | DELETE 请求 | 数据清空、标志设置 |

#### 9.4.2 测试组织原则

1. **AAA 模式**：Arrange（准备）→ Act（执行）→ Assert（断言）

3. **可读性**：测试代码应该像文档一样易读
4. **可维护性**：避免脆弱的测试，使用数据属性选择器

### 10.2 常见问题解答

**Q1: 为什么选择 Vitest 而不是 Jest？**

A: Vitest 与 Vite 无缝集成，配置简单，执行速度更快，且原生支持 TypeScript。

**Q2: 为什么 E2E 测试由 AI 手动执行而不是自动化？**

A: 个人项目规模较小，AI 通过 DevTools 执行 E2E 测试更灵活，且可以验证视觉和交互细节。

**Q3: 如何保证测试的可靠性？**

A: 通过合理的测试清理机制、独立的测试数据、以及用户视角的测试方法。

**Q4: 测试覆盖率应该达到多少？**

A: 目标 85%+，但更重要的是测试关键路径和核心功能，而非盲目追求数字。
